# Gemma 4 概要と特性

## Gemma 4 とは

Gemma 4 は **Google DeepMind** が2025年4月にリリースしたオープンウェイトの大規模言語モデルファミリーである。Gemini の技術をベースに、研究者・開発者が自由に利用・改変できるよう設計されている。

マルチモーダル対応（テキスト＋画像入力）を持つVLMバリアントを含み、オンプレミス環境での運用が可能である点が特徴である。

---

## アーキテクチャ特性

### マルチモーダル対応

- テキストと画像の両方を入力として受け付ける
- Vision Encoder（ViT ベース）により画像を視覚トークンに変換し、Language Model に統合する
- 複数画像の同時入力にも対応している

### 利用可能なモデルサイズ

# 利用可能なモデルサイズ（Gemma 4系）

| モデル | パラメータ規模 |推論時実行パラメータ |動作条件（目安） | 特性 |
|--------|----------------|-----------------|------------------|------|
| E2B | 約5.1B（総パラメータ）|約2B | スマホ / 低RAM PC / エッジデバイス | 超軽量、低消費電力、精度は限定的 |
| E4B | 約8.0B（総パラメータ）|約4.5B | 8〜16GB RAM / ノートPC | バランス型、実用レベルの精度 |
| 26B（MoE） | 約260億（MoE構成）|約3.8B | RTX 4090 / 5090（24–32GB VRAM） | 高性能・効率的推論（エージェント向け） |
| 31B（Dense） | 約310億（Dense）| 30.7B(全パラメータ使用)| A100 / H100 / 32GB以上VRAM | 最高品質、安定した推論性能 |

---

## 補足

RTX 5090（32GB VRAM）環境では：
- **26B（MoE） → 量子化（Q4/Q5）で快適にファインチューニング可能**
- **31B（Dense） → QLoRA（4bit）でギリ実用レベル（VRAM最適化が必須）**

### コンテキスト長

- 最大128K or 256K トークン（モデルバリアントによる）の長文対応
- 長い図面の説明文、複数ページのドキュメント解析に対応
- 高解像度画像を多数のビジュアルトークンで表現可能

## 成功事例（応用領域）

### ドキュメント解析・OCR

- PDF、スキャン画像からのテキスト抽出
- 表・フォームの構造化データへの変換
- 多言語ドキュメントの読み取りと翻訳

### 医療画像の説明生成（研究段階）

- X線・MRI画像の所見記述支援
- 病変候補領域の言語的説明
- 医療レポートの自動ドラフト生成（人間によるレビュー前提）

### UI/スクリーンショット理解

- Webページ・アプリのスクリーンショットから要素を特定
- UIテストの自動化支援
- アクセシビリティ向上のための画面説明生成

### 工業図面・技術文書の読み取り（emerging）

- 電気図面、機械図面の部品・接続関係の抽出
- 部品番号・仕様値の読み取り（OCR的用途）
- BOM（部品表）の自動生成支援
- **本プロジェクトが取り組む「コネクタ検出」もこの領域に位置する**

### コード生成補助

- スクリーンショットやUIモックからのコード生成
- 図解から実装コードへの変換

## オープンウェイトのメリット（ライセンス）

Gemma 4 は **Gemma Terms of Use** のもとで提供されており、以下の用途が許可されている。

- 商用利用（条件あり、詳細は公式ライセンスを確認）
- モデルの改変・ファインチューニング
- オンプレミス環境でのデプロイ（クラウドAPI不要）
- 派生モデルの配布

### オンプレ運用での主なメリット

| メリット | 説明 |
|---|---|
| データプライバシー | 機密図面データを外部に送信しない |
| レイテンシ | ネットワーク遅延なしで低遅延推論 |
| コスト | API課金なし、ハードウェアのみのコスト |
| カスタマイズ | ファインチューニングで自社データに特化可能 |
| 可用性 | インターネット接続不要、工場内LAN環境でも稼働 |

## Gemma 4 の得意なこと・苦手なこと

### 得意なこと

| カテゴリ | 具体的な能力 |
|---|---|
| 自然言語理解 | 複雑な指示の解釈、文脈を踏まえた回答生成 |
| 画像の内容説明 | 「何が写っているか」を詳細に記述 |
| OCR的な読み取り | 画像内のテキスト・数値の抽出 |
| 構造化データ出力 | JSON / CSV 形式での回答生成（プロンプト設計次第） |
| 複数画像の比較 | 複数枚の図面を比較して差異を説明 |
| 多言語対応 | 日本語を含む多言語での入出力 |

### 苦手なこと

| カテゴリ | 説明 | 対策 |
|---|---|---|
| 精密な座標出力 | bounding box の座標が不正確になることがある | ファインチューニングで改善 |
| リアルタイム処理 | 推論時間が数秒〜数十秒かかる | バッチ処理設計、またはYOLOとのハイブリッド |
| 微細なシンボル識別 | 図面の細かいシンボルを見落とすことがある | 高解像度入力、画像の分割処理 |
| 一貫した出力形式 | プロンプトが弱いと出力フォーマットがばらつく | 出力形式をプロンプトで厳密に指定 |
| 絶対的な計数精度 | 「コネクタが何個あるか」の正確な計数は難しい | ファインチューニングまたは補完的手法との組み合わせ |

## 図面からのコネクタ抽出への適用可能性

### ゼロショット（学習なし）での適用

即座に試すことができる。品質は限定的だが、プロトタイピングに有効である。

```
プロンプト例:
「添付の電気図面を解析してください。
図面内に存在するコネクタ（接続端子、プラグ、ソケット等）をすべて特定し、
以下のJSON形式で出力してください:
{
  "connectors": [
    {"id": 1, "type": "コネクタ種類", "description": "場所や特徴の説明"}
  ]
}」
```

**期待精度**: 50〜70%程度（ドメイン特化していない状態）

### ファインチューニング後の適用（推奨）

図面コネクタ検出の精度を本番レベルに引き上げるには、アノテーション済みデータによるファインチューニングが推奨される。

- 最低100〜500枚のアノテーション済み図面で精度が大幅に向上?
- 出力フォーマット（JSON + 座標）の一貫性が確保される
- 工場固有のコネクタ種類を学習させることが可能

**期待精度**: 85〜95%程度（データ品質・量による）

### 出力形式をJSON（座標、種類）で指定するプロンプト設計

精密な出力を得るためには、**プロンプト内で出力スキーマを明示的に定義**することが重要である。

```
「以下の形式でのみ回答してください。説明文は不要です。

{
  "connectors": [
    {
      "id": <整数>,
      "type": "<terminal|plug|socket|header|その他>",
      "bbox": [<x1>, <y1>, <x2>, <y2>],
      "part_number": "<部品番号またはnull>",
      "confidence": <0.0〜1.0>
    }
  ],
  "total_count": <整数>
}

座標は画像の左上を(0,0)とし、右下を(width, height)とするピクセル座標で示してください。」
```

### 推奨アーキテクチャ（本プロジェクト向け）

```
入力図面画像
    ↓
前処理（解像度調整、コントラスト強調）
    ↓
Gemma 4 26B (QLoRA ファインチューニング済み)
    ↓
JSON出力（コネクタID、種類、座標、部品番号）
    ↓
後処理（座標検証、重複除去、信頼度フィルタリング）
    ↓
検出結果
```
